# AI-Blog 自動改善システム 全体設計（plan.md）

最終更新: 2026-07-19

## 目的

サイトの閲覧データ（GA4）を分析し、AIが自ら改善案を立てて実装・公開し、
その学習過程をファイルに蓄積しながら改善を繰り返す仕組みを作る。
壊れた場合は自動で検知して直前の状態に戻せる安全網を持つ。

## 全体像（3フェーズ）

| フェーズ | 内容 | 状態 |
|---|---|---|
| フェーズ1 | 計測基盤（GA4導入） | ✅ 完了（2026-07-19、測定ID: G-6NXKRMQDZ0） |
| フェーズ2 | データ取得 → 分析 → 学習記録 → 改善実装 | 🔨 これから構築 |
| フェーズ3 | 安全網（検証・自動ロールバック・監視役） | 🔨 これから構築 |

## 運用の前提

- 実行枠: 既存の祭り情報ルーティン（1日5回: 08:00 / 12:00 / 16:00 / 17:00 / 18:00）の中に
  改善ステップを組み込む。改善専用のルーティンは作らない（Pro枠の制約のため）。
- 改善対象: `scripts/generate_html.py`（HTMLテンプレート・CSS・ページ構成）のみ。
- 改善対象外（AIに触らせない）: `scripts/post_instagram.py`, `scripts/git_push.py`,
  `scripts/save_data.py`, `config.json`, `festival-routine-prompt.md`, ルーティン設定, 環境変数。
- データが薄い期間（訪問者が少ない間）は分析結果も薄いが、パイプラインを先に動かしておき、
  データが増えた時点から本格的な学習が回る状態にする。

## フェーズ2: データ取得・分析・学習・改善

### ①データ取得層: `scripts/fetch_analytics.py`（新規）

- GA4 Data API からサイトの閲覧データを取得する。
- 取得する指標（初期セット）:
  - ページ別: ページビュー数、平均エンゲージメント時間、直帰率相当
  - サイト全体: セッション数、流入元（organic / social / direct）、デバイス比率
- 取得結果は `data/analytics_snapshot.json` に保存（実行日時付き）。
- 認証: GA4 のサービスアカウント（JSONキー）を使用。キーはルーティンの環境変数に登録し、
  リポジトリには絶対にコミットしない。
- 実行タイミング: 1日1回で十分（改善ステップが走る回のみ取得）。

### ②分析・学習層: ルーティンの改善ステップ（プロンプト拡張）

- 祭り情報ルーティンのプロンプトに「改善ステップ」を追加する。
- 改善ステップの流れ:
  1. `data/analytics_snapshot.json`（最新の閲覧データ）を読む
  2. `data/learning_log.json`（過去の学習記録）を読む
  3. 分析: 離脱が多いページ、読まれていないページ、流入の弱さ等を特定
  4. 改善案を1つだけ立てる（1回の実行で1改善。小さく確実に）
  5. `scripts/generate_html.py` を修正して実装
  6. フェーズ3の検証を通してから push
  7. 実行した改善・理由・期待効果・結果測定予定日を `data/learning_log.json` に追記
- 学習記録 `data/learning_log.json` のスキーマ:

```json
{
  "entries": [
    {
      "date": "YYYY-MM-DD HH:MM",
      "analysis": "データから読み取った課題",
      "action": "実施した改善の内容",
      "target_files": ["scripts/generate_html.py"],
      "hypothesis": "この改善で何がどう良くなるはずか",
      "verify_after": "YYYY-MM-DD（効果測定予定日）",
      "result": "効果測定後に追記（改善した/変化なし/悪化した）",
      "commit": "コミットハッシュ"
    }
  ]
}
```

- 過去の entries を必ず読み、同じ失敗を繰り返さない・成功パターンを再利用する。
- データが不十分な期間の改善テーマ（アクセスゼロでも意味があるもの）:
  - SEO基礎: 各ページの title / meta description の充実
  - サイトマップ (sitemap.xml) の生成と Search Console 登録準備
  - 構造化データ (JSON-LD, Event スキーマ) の追加
  - 表示速度・アクセシビリティの改善

### 改善頻度の制御

- 毎回の実行で改善するのではなく、1日1回（08:00の回のみ）改善ステップを実行する。
  他の4回は従来通り祭り情報の更新のみ。
- 理由: 改善のたびにサイトが変わると効果測定ができない。変更は小さく、間隔を空けて、
  効果を測ってから次へ。

## フェーズ3: 安全網

### ③検証スクリプト: `scripts/validate_site.py`（新規）

push 前に必ず実行し、全チェックが通らなければ push しない:
- `python scripts/generate_html.py` がエラーなく完走する
- 生成された docs/ 配下の全 HTML が構文的に壊れていない（パース可能）
- 内部リンク切れがない（prefecture/ month/ への遷移が全て存在するファイルを指す）
- GA4 タグが全ページに入っている
- トップページに祭りカードが1件以上表示されている

### 自動ロールバック

- 改善 push の後、次回ルーティン実行の冒頭で「前回の改善後にサイトが壊れていないか」を
  `validate_site.py` で再確認する。
- 壊れていた場合: 改善コミットを `git revert` で打ち消す（force push は使わない）。
  revert したことを `data/learning_log.json` の該当 entry の result に記録する。

### 監視役（別役割のAI）

- 改善を実装する役と、検証する役を同一実行内で分離する:
  - 実装役: 改善ステップで generate_html.py を修正する
  - 監視役: 修正後、実装の意図を知らない前提で `validate_site.py` と目視相当のチェック
    （生成HTMLの主要ページを読み、表示崩れ・意味破壊がないか）を行い、
    問題があれば push を拒否して差し戻す
- 将来、API 経由の別ルーティン等に監視役を分離することも検討（現状は Pro 枠の制約で同一実行内）。

### 絶対的な禁止事項（安全網の根幹）

- `git push --force` / `--force-with-lease` の使用禁止
- 改善対象外ファイルの変更禁止
- 1回の改善で複数の大改造をしない（1実行1改善）
- learning_log.json への記録なしに改善を push しない

## 実装順序（これからやること）

1. GA4 サービスアカウント作成・Data API 有効化・環境変数登録（人間の作業）
2. `scripts/fetch_analytics.py` 実装・テスト
3. `scripts/validate_site.py` 実装・テスト
4. `data/learning_log.json` 初期化
5. ルーティンプロンプトに改善ステップ（08:00 のみ発動）を追記
6. 数日間観察し、learning_log が正しく蓄積されるか確認
7. データが溜まってきたら分析の質を評価・チューニング
